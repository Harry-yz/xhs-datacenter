## 1. Summary
本 PRD 定义橙果视界 XHS 数据中心的长期产品规范。  
目标是让用户在搜索品牌、品类、达人时，稳定、快速看到结果；当库内数据不足时，系统自动补采并自动刷新，不需要整页刷新。  
文档同时定义数据资产与版本路线，作为后续版本更替的统一依据。

## 3. Background
### 3.1 Context
项目已形成完整架构：`Next.js + FastAPI + PostgreSQL + Redis + Celery + Docker + OpenResty`。  
当前已具备首页六卡片、搜索工作台、任务补采、回调入库、轮询更新等基础能力。

### 3.2 Why Now
过去阶段出现过影响体验的问题：搜索卡慢、命中误 pending、页内换词不更新、补采后不自动展示、偶发 502。  
这些问题会直接影响客户演示与业务转化，所以要从“临时修复”升级到“长期稳定架构”。

### 3.3 What Became Possible
项目已完成关键基础：  
- Web 容器守护启动与健康检查  
- 搜索 `v2 + recall guard + fallback`  
- 搜索任务轮询与去重机制  
- 增量刷新链路（term/author/change log）

现在可以把能力标准化到主 PRD，持续演进。

## 4. Objective
### 4.1 总目标
打造一个“稳定、快速、可解释”的 XHS 数据中心，让客户可以放心用于日常分析与对外演示。

### 4.2 价值目标
- 对客户：搜索更快，结果更准，操作更顺滑。  
- 对公司：演示成功率和转化率更高。  
- 对团队：减少线上救火，形成长期可维护规范。

### 4.3 Key Results（SMART）
**O1 稳定可用**
- KR1：库内命中首屏响应 `P95 <= 1.2s（品类/品牌）`，`<= 1.5s（达人）`（4 周）
- KR2：无命中首包 pending 返回 `P95 <= 600ms`（4 周）
- KR3：连续 30 分钟压测 `502 = 0`（持续）

**O2 搜索正确**
- KR1：热词（如 `YSL`、`防晒`、`洗发水`）`status=ready 且 items>0` 成功率 `>= 99%`（4 周）
- KR2：排序点击可见重排成功率 `>= 99.5%`（3 周）
- KR3：页内换词后旧结果残留率 `< 0.5%`（3 周）

**O3 补采可靠**
- KR1：冷词 pending 后 60 秒内出现首批结果成功率 `>= 95%`（4 周）
- KR2：同词重复点击产生重复补采任务比例 `< 2%`（4 周）
- KR3：超时 running 自动回收率 `>= 99%`（4 周）

## 5. Market Segment(s)
按用户要完成的“工作任务”分段：

### Segment A：品牌主与市场负责人
- 任务：判断品牌/品类机会，做投放与内容策略。
- 约束：希望快速看到可解释结果，不接受卡住和空白页。

### Segment B：代理商与运营团队
- 任务：高频切词、切排序、翻页对比，输出执行方案。
- 约束：要求页内交互稳定，结果不能串台。

### Segment C：售前顾问与演示团队
- 任务：在客户会中稳定演示搜索与洞察能力。
- 约束：可用性优先，不能出现 502、跳转错乱、补采卡死。

## 6. Value Proposition(s)
### 6.1 用户要完成的工作
- 输入关键词后快速拿到相关内容。
- 没有数据时自动补采，并在页面内自动更新。
- 不只看列表，还能看到可行动的爆点和痛点。

### 6.2 用户获得的收益
- 决策更快：先看现有结果，再自动补齐。
- 演示更稳：减少卡顿、空页、手动刷新。
- 复盘更清晰：结果语义和状态一致。

### 6.3 用户避免的痛点
- “明明有数据却显示没有结果”
- “点击排序/翻页没反应”
- “pending 卡住，必须刷新整个浏览器”
- “重复补采导致队列拥堵”

### 6.4 与替代方案的差异
- 数据命中优先展示：更快  
- 无命中自动补采：更完整  
- 页内状态机保护：更稳定  
- 任务去重与守护部署：更可运维

## 7. Solution
### 7.1 UX / Prototypes（核心流程）
1. 未登录：入口、行业卡片、结果页都先弹登录。  
2. 已登录搜索：库内有数据直接 `ready` 展示。  
3. 无命中：返回 `pending + job_id`，前端显示骨架并自动轮询。  
4. 补采达到首批结果后：页内自动替换为结果。  
5. 排序、翻页、换词：都在页内完成，旧请求不覆盖新结果。

### 7.2 Key Features
#### F1. 登录门禁（硬约束）
- 入口、行业卡片、结果页未登录访问必须触发登录流程。  
- 保留 `q/type/page/sort/order` 回跳参数。

#### F2. 搜索语义固定
- `hit=true`：返回 `ready`，先展示已有内容。  
- `hit=false`：返回 `pending`，后台触发或复用补采任务。  
- `failed`：展示轻提示，不显示误导性“空结果即结束”。

#### F3. 补采与轮询
- 补采按 `query + type + filters` 去重。  
- 前端自动轮询并在就绪后自动更新。  
- 当前页不足 30 条时允许后台继续补采，不阻塞当前展示。

#### F4. 排序与分页
- 排序触发强一致请求，旧请求结果丢弃。  
- 支持数字分页与页内切换。  
- 维持 `has_more` 语义，减少重型计数阻塞。

#### F5. 守护可用性
- Web 走容器守护与健康检查。  
- 网关反代保持稳定。  
- 发布设闸门：关键搜索接口必须 200 且热词可 ready 返回。

### 7.3 Technology（当前实现基线）
- 前端：Next.js App Router，页内状态机（abort/requestVersion/poll）
- 后端：FastAPI 搜索与任务 API，`v2 + recall guard + fallback`
- 数据：PostgreSQL（事实/维度/关系/任务）
- 异步：Celery + Redis（补采、回调、增量刷新、回收）
- 运维：Docker Compose + OpenResty + 守护脚本

### 7.4 Assumptions（待持续验证）
- 每页固定 30 条可覆盖主要场景。  
- 默认时间窗（近 30 天）可平衡时效与样本量。  
- 补采去重与限流可持续控制灰豚额度风险。  
- 登录门禁是长期不变规则，不允许移除。

## 8. Release
### V1（稳定基线，1-2 个迭代）
- 登录门禁、搜索语义、补采闭环、排序分页、守护部署全部稳定
- 发布闸门与巡检脚本常态化

### V1.5（深化分析，1-2 个迭代）
- 在搜索结果基础上增加更强的分析视图（爆点/痛点模块）
- 优化行业和品牌口径的一致性

### V2（智能洞察，持续迭代）
- 更自动化的洞察抽取与报告化输出
- 更细粒度的人群、行业、内容对比分析
