## 1. 项目背景

小红书已成为品牌种草、内容运营和用户口碑管理的重要阵地。品牌和运营团队在日常工作中需要频繁回答以下问题：

- 某个品类、品牌或关键词在小红书上的内容规模和讨论热度如何？
- 有哪些高互动笔记值得研究？
- 有哪些达人值得合作？
- 某个行业最近 7/30/90 天的内容趋势如何变化？
- 已搜索过的词能否快速复用结果，而不是重复等待？

传统做法往往依赖多个分散脚本、人工导表和零散查询，存在以下问题：

- 搜索体验不稳定，命中过的数据也可能需要重新等待
- 新词检索耗时长，用户缺少明确的任务状态反馈
- 数据中心页面、搜索页、详情页之间缺少统一的数据结构
- 采集、回调、入库、搜索、看板之间缺少工程化闭环
- 线上故障容易表现为“页面崩溃”“只剩文字”“搜索超时”等用户可见问题

XHS Data Center 的目标，就是搭建一个统一的数据分析工作台，把“小红书内容洞察”从零散工具升级为稳定、可复用、可扩展的产品系统。

## 2. 产品愿景

构建一个面向品牌、运营和分析团队的小红书数据中心平台，实现：

- 老词秒开：优先复用数据库与缓存中的已有结果
- 新词可等：首请求立即返回任务状态，而不是无反馈卡死
- 结果可追：搜索过程具备 pending / ready / failed 的明确状态机
- 数据可沉淀：采集结果写入统一数据模型，支持后续看板、详情、报告复用
- 系统可运营：具备灰度、监控、健康检查、回滚和稳定性治理能力

## 3. 产品目标

### 3.1 业务目标

- 为品牌和内容团队提供统一的小红书数据工作台
- 提升搜索、看板和详情分析的使用效率
- 将采集数据沉淀为长期可复用的数据资产
- 支持品类、品牌、达人、笔记等多维分析
- 为后续报告生成、策略分析和运营决策提供底座

### 3.2 用户体验目标

- 已搜索过的关键词，优先即时展示已有结果
- 新词搜索不再“长时间白等”，而是立即进入任务态并自动刷新
- 页面在上游字段缺失、接口超时、异步处理中也不崩溃
- 首页和搜索页都具备稳定、清晰、可预期的交互反馈

### 3.3 工程目标

- 建立“前端展示层 + BFF 代理层 + FastAPI 业务层 + Celery 异步层 + PostgreSQL 数据层”的清晰边界
- 降低线上故障对用户的可见影响
- 提高搜索链路的稳定性和可恢复能力
- 保证静态资源、前端 hydration、API 代理、任务调度链路具备健康检查能力

## 5. 核心问题定义

本项目重点解决 4 类问题：

### 5.1 搜索效率问题

- 老词搜索重复等待，用户感知差
- 新词搜索耗时长，没有清晰反馈
- 结果完整度和首屏速度难以兼顾

### 5.2 数据孤岛问题

- 搜索、看板、详情页没有统一的数据模型
- 数据采集结果难以复用于多个分析场景

### 5.3 系统稳定性问题

- 前端静态资源异常会导致页面只剩文字或 hydration 失败
- 后端请求路径若混入重操作，容易出现锁等待或超时
- Redis、磁盘、任务系统异常会直接影响用户搜索体验

### 5.4 可运营性问题

- 缺少统一的健康检查
- 缺少前端上线后的 smoke test
- 缺少线上守护与快速回滚流程

## 7. 关键产品策略

### 7.1 搜索策略：缓存优先 + 增量补采

#### 命中已有数据

- 优先返回数据库 / Redis 中已有结果
- 页面立即展示已有列表
- 后台继续补采最新数据
- 前端进入 `pending` 状态并自动轮询

#### 未命中已有数据

- 立即创建异步搜索任务
- 返回 `pending + job_id`
- 一旦采集到可展示结果，页面自动刷新
- 最终任务完成后进入 `ready`

### 7.2 稳定优先的结果更新策略

- 首屏尽量稳定，不让已展示结果频繁抖动
- 新增数据优先作为增量补充
- 页面自动刷新，但尽量保持用户已见排序感知

### 7.3 DB-first 超时降级策略

- 新词第一次搜索时，数据库首查设置短超时
- 一旦超过阈值，不再继续阻塞用户请求
- 直接降级为“未命中，转异步任务”
- 避免新词在前端代理层表现为整体超时

## 8. 用户流程

### 8.1 首页进入

1. 用户访问数据中心首页
2. 查看六卡片总览、趋势和行业矩阵
3. 在搜索框输入关键词
4. 选择品类/品牌或达人类型
5. 发起搜索

### 8.2 老词搜索

1. 用户输入已搜索过的关键词
2. 前端调用搜索代理接口
3. 后端优先命中缓存或数据库快照
4. 页面立即展示结果
5. 后台启动补采任务
6. 页面进入 `pending` 并自动刷新
7. 补采完成后页面更新为最新结果

### 8.3 新词搜索

1. 用户输入未搜索过的关键词
2. 前端调用搜索代理接口
3. 后端 DB-first 首查短超时
4. 若未命中或超时，则直接创建异步任务
5. 页面进入 `pending`
6. 轮询 job 状态
7. 采集到可展示数据后刷新页面
8. 最终完成后进入 `ready`

## 12. 数据流设计

### 12.1 搜索数据流

1. 用户在前端输入关键词
2. Next.js API Route 转发到 FastAPI 搜索接口
3. FastAPI 先读 Redis 结果缓存
4. 未命中再走 PostgreSQL DB-first 查询
5. 若结果不足，创建 Celery 任务
6. worker 发起外部采集
7. callback 入库
8. 前端轮询任务接口并刷新结果

### 12.2 看板数据流

1. 前端请求看板接口
2. 后端优先读 Redis / 内存缓存
3. 未命中再执行 SQL 聚合查询
4. 返回总览指标与趋势数据

## 13. 非功能需求

### 13.1 性能要求

- 老词搜索首屏尽量秒开
- 新词搜索避免整体同步等待
- 看板高频接口应具备缓存能力

### 13.2 稳定性要求

- 静态资源异常不能被误判为站点正常
- 搜索 pending 状态要有明确 fallback
- 请求路径不允许引入高风险 DDL 或长锁等待

### 13.3 可维护性要求

- 前后端边界清晰
- 支持灰度配置
- 支持脚本化运维
- 支持健康检查与回滚

### 13.4 可观测性要求

- API 日志可区分 timeout / upstream error / misconfigured
- 发布后支持静态资源 smoke test
- 搜索任务具备可追踪 job_id

## 14. 已落地的关键稳定性治理

本项目不仅是一个业务产品，也体现出较强的线上治理能力，已经落地的治理方向包括：

- 搜索结果归一化，避免页面因字段异常而崩溃
- 搜索页状态机治理，避免轮询和旧请求回写导致 UI 抖动
- 登录弹窗 provider 稳定化，减少重复 effect 触发
- 静态资源健康检查，修复 HTML 正常但 CSS/JS 500 的上线事故
- 后端请求路径 bootstrap 风险治理，避免 DDL 导致锁等待
- Redis 磁盘满导致写保护时的故障排查与恢复
- 新词 DB-first 查询超时降级，避免搜索冷启动超时

## 17. 面试可直接使用的项目介绍

XHS Data Center 是一个面向小红书场景的数据分析平台，我负责的重点是搜索链路、前后端代理层和系统稳定性治理。这个系统前端基于 Next.js 和 React，后端基于 FastAPI，主数据存储在 PostgreSQL，异步采集由 Celery + Redis 驱动。项目的核心设计是“数据库优先查询 + 异步补采”，让老词搜索能够优先展示已有结果，让新词搜索快速进入任务态并自动刷新。除了业务功能本身，我还处理了静态资源 500、Redis 因磁盘满写保护、请求路径锁等待和新词搜索超时等线上问题，因此这个项目既能体现产品设计能力，也能体现工程治理和问题排查能力。
